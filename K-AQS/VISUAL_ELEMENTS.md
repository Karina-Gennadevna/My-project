# K-A-Q-S™ Visual Elements — UX Blueprint
_Версия 1.0 — 2026-03-06_

---

## Общие принципы для всех визуальных элементов

Каждый элемент — это **не украшение, а инструмент**. Он должен передавать информацию или создавать ощущение системности. Референс: Palantir Foundry data visualizations, Bloomberg Terminal charts, Apple keynote diagrams.

**Визуальный язык всех элементов:**
- Тёмный фон, тонкие светящиеся линии
- Оранжевый — только для данных и акцентов, не для фона
- IBM Plex Mono для всех числовых меток
- Никаких drop shadows на самих визуализациях — только на контейнерах
- Штрихи и линии — не толще 1.5px, кроме акцентных элементов
- Все данные выглядят живыми — как будто обновляются в реальном времени

## 1. Главная диаграмма системы K-A-Q-S™

### Задача
Объяснить систему как единое целое: четыре измерения не существуют сами по себе — они пересекаются и влияют друг на друга. Это не список, это **архитектура**.

### Где используется
- Экран 4 («Система K-A-Q-S™») — как центральный визуальный элемент
- Может появиться уменьшенной версией в Hero как фоновая деталь

### Композиция

Центральная схема с четырьмя квадрантами, соединёнными в единое ядро.

```
        K
        |
   ─────●─────
   |    |    |
Q ─●────◉────●─ A
   |    |    |
   ─────●─────
        |
        S
```

**Центральный элемент (◉):** кольцо с текстом «УПРАВЛЯЕМОСТЬ» или просто символ K-A-Q-S™. Это точка пересечения всех четырёх осей. Диаметр: 48–56px. Стиль: двойное кольцо, внешнее — тонкое с `border-accent`, внутреннее — заполнено `accent-glow`.

**Четыре узла (●):** равноудалённые от центра по диагоналям. Каждый узел — окружность диаметром 32px с буквой (K/A/Q/S), `bg-elevated`, border `border-accent`.

**Оси:** тонкие линии от узлов к центру, `rgba(249,115,22,0.3)`, 1px. Не горизонтально/вертикально — по диагонали (45°). Это создаёт ромбовидную форму, более динамичную чем крест.

**Подписи узлов:** рядом с каждым узлом — полное название на русском в 2 строки. `body-sm`, `text-secondary`. Расположение: K — вверх-слева, A — вверх-справа, Q — низ-слева, S — низ-справа.

**Внешние кольца:** 2 концентрических окружности вокруг всей схемы. `border-subtle`, `rgba(255,255,255,0.04)`. Первое кольцо — плотно, второе — с gap. Имитируют «поле охвата» системы.

**Связи между узлами (не через центр):** пунктирные дуги K↔A, A↔S, S↔Q, Q↔K. `rgba(255,255,255,0.08)`, `stroke-dasharray: 3 5`. Показывают взаимозависимость измерений.

### Цвета и состояния

| Состояние | Визуал |
|----------|--------|
| Default | Все элементы в базовой яркости |
| Hover на узел K | Узел K подсвечивается, линии к K → `accent-primary`, подпись → `text-primary`, остальные затемняются до 40% opacity |
| Hover на центр | Все линии разгораются одновременно, центральное кольцо пульсирует один раз |

### Анимации

1. **Появление (одноразовое, при scroll):** центр появляется первым (scale 0→1, 400ms), затем оси «рисуются» от центра к узлам по очереди (stroke-dashoffset animation, 600ms, stagger 120ms), затем подписи fade-in (400ms, stagger 80ms).

2. **Idle (бесконечная, очень тихая):** центральное кольцо медленно вращается — только внешнее из двух, 60 секунд на полный оборот. Едва заметно.

3. **Hover:** 200ms ease-out, без scale.

### Премиальность

- Метки осей в IBM Plex Mono
- Центральный символ содержит крошечную точку в середине (как перекрестие прицела, как радар-центр)
- Штрих осей на концах переходит от сплошного к прозрачному (gradient stroke)
- Общий размер схемы на десктопе: ~480×480px. На мобиле: ~280×280px

## 2. Radar Chart — четыре оси управляемости

### Задача
Показать результат диагностики: где сильные стороны, где зоны риска. Главная визуализация в отчёте и мокапе на лендинге.

### Где используется
- Экран 6 («Диагностика») — как мокап результата, правая колонка
- Страница результатов платформы — как главный виджет

### Композиция

Классическая паутина (radar/spider chart) с четырьмя осями: K, A, Q, S.

**Структура:**
- Центр координат — точка (0,0)
- 4 оси расходятся из центра под углом 90° (как крест или как ромб с поворотом 45°)
- 5 концентрических многоугольников (уровни 1–5) — сетка
- Заполненная область — результат компании
- Точки данных на каждой оси — отмечены кружком

**Сетка (паутина):**
- 5 уровней = 5 многоугольников
- Линии сетки: `rgba(255,255,255,0.05)`, 0.75px
- Оси сетки (от центра к краям): `rgba(255,255,255,0.08)`, 1px

**Заполненная область (данные компании):**
- Fill: `rgba(249,115,22,0.12)` — тёплый оранжевый полупрозрачный
- Stroke: `accent-primary`, 1.5px
- Точки пересечения: кружок диаметром 5px, fill `accent-primary`, stroke `bg-base` 2px

**Метки осей:**
- K / A / Q / S — IBM Plex Mono, 11px, uppercase, `text-muted`
- Числовые метки уровней (1–5) на одной из осей: IBM Plex Mono, 10px, `text-muted` opacity 0.5
- Полные названия под буквами: Inter, 10px, `text-secondary`, max-width 80px, text-align center

**Бенчмарк (опционально):**
- Вторая заполненная область — медиана по рынку
- Fill: `rgba(255,255,255,0.04)`, stroke: `rgba(255,255,255,0.15)`, stroke-dasharray: 4 4
- Легенда: два цветных квадрата + подписи, `caption` Inter

### Цвета и состояния

| Состояние | Визуал |
|----------|--------|
| Default | Стандартная заливка, точки данных статичны |
| Hover на точку данных | Точка увеличивается до 8px, всплывает tooltip с точным значением и названием измерения |
| Hover на ось | Подсвечивается вся ось, соответствующая точка данных пульсирует |

**Tooltip:** `bg-elevated`, border `border-default`, padding 8px 12px, border-radius 6px. Содержит: название измерения (body-sm, text-primary), значение (IBM Plex Mono, 18px, text-accent), краткую интерпретацию (caption, text-secondary).

### Анимации

1. **Появление:** Заполненная область «вырастает» из центра — fill и stroke рисуются одновременно от центра к периметру. 800ms, ease-out. Точки данных появляются в конце с scale 0→1, stagger 100ms.

2. **Счётчики рядом с диаграммой:** если есть числовые показатели — count-up анимация синхронно с диаграммой.

3. **Интерактивность:** hover-состояния с 150ms ease-out.

### Премиальность

- Слабый radial gradient от центра к краям в качестве фонового свечения: `radial-gradient(circle, rgba(249,115,22,0.05), transparent 70%)`
- Тени под точками данных: `0 0 6px rgba(249,115,22,0.5)`
- Размер: 320×320px в мокапе на лендинге, 400×400px на странице результатов
- Рядом с диаграммой — 4 числовых показателя в колонку: буква (IBM Plex Mono, 24px, accent), название, значение X.X/5.0

## 3. Risk Map — карта операционных рисков

### Задача
Визуализировать выявленные риски по двум осям: вероятность и критичность. Дать руководителю понять, что именно угрожает управляемости прямо сейчас.

### Где используется
- Экран 6 («Диагностика») — как второй элемент мокапа результата, под radar chart
- Страница результатов — как отдельный раздел отчёта

### Композиция

**Матрица 3×3 или 4×4** (как Heat Map / Impact-Probability Matrix).

Оси:
- Горизонтальная X: «Вероятность» (Низкая → Высокая)
- Вертикальная Y: «Критичность» (Низкая → Высокая)

**Зоны матрицы (градиент фона):**
- Нижний левый угол (низкая вероятность, низкая критичность): `rgba(34,197,94,0.05)` — зона принятия
- Центр и правый верхний угол: переход через `rgba(234,179,8,0.07)` — зона мониторинга
- Правый верхний угол (высокая вероятность, высокая критичность): `rgba(239,68,68,0.1)` — зона действия

Переходы зон — не жёсткие границы, а плавный радиальный градиент от угла к углу.

**Риски (точки/кружки на матрице):**
- Каждый риск — круг диаметром 24–36px (размер зависит от «силы» риска или его веса)
- Цвет заливки: по зоне — зелёный/жёлтый/красный, но приглушённый (opacity 0.7)
- Stroke: 1px, тот же цвет opacity 1
- Внутри круга — номер или индекс риска (IBM Plex Mono, 10px, белый)

**Метки рисков:**
- Рядом с каждым кружком — строка текста с названием риска
- `caption`, `text-secondary`, max 2 строки
- Если места мало — метка только в tooltip

**Оси:**
- Тонкие линии, `border-subtle`
- Подписи осей: `label` Inter, uppercase, `text-muted`
- Тики осей: маленькие штрихи через равные промежутки

**Легенда:** внизу или справа от матрицы. 3 строки: Принятие / Мониторинг / Действие — с цветовыми квадратами.

### Цвета и состояния

| Состояние | Визуал |
|----------|--------|
| Default | Все точки видны, зонировый gradient фон |
| Hover на точку | Точка увеличивается, всплывает tooltip с: названием риска, описанием, рекомендуемым действием |
| Hover на зону | Зона слегка подсвечивается, в легенде активируется соответствующая строка |

### Анимации

1. **Появление:** сетка матрицы fade-in (300ms), зональный gradient разгорается (500ms), точки рисков падают с верха с ease-bounce-подобным settle (не классический bounce, а просто небольшой overshoot на 4px), stagger 80ms между точками.

2. **Hover:** 150ms ease-out, tooltip fade-in 100ms.

### Премиальность

- Фон матрицы — не чёткие цветные зоны, а едва различимый тепловой градиент. Как у Bloomberg risk dashboards.
- Крупные риски перекрывают мелкие — это нормально и добавляет реализма
- Номера рисков ведут к пронумерованному списку ниже карты
- Размер: 480×360px на десктопе

## 4. Evolution Map — карта прогресса управляемости

### Задача
Показать динамику: как менялась управляемость компании во времени. Инструмент для подписчиков платформы — видеть прогресс после каждой диагностики.

### Где используется
- Экран 8 («Платформа») — как аргумент в пользу подписки. Мокап показывает «как выглядит прогресс за 3 месяца».
- Дашборд платформы — основной виджет истории

### Композиция

**Multi-line chart** — временная шкала с 4 линиями (по одной на K/A/Q/S).

Структура:
- Горизонтальная ось X: временная шкала (месяцы или даты оценок)
- Вертикальная ось Y: значение управляемости (0–5 или 0–100)
- Четыре линии данных — по одной на каждое измерение
- Точки оценок на каждой линии — квадратные или ромбовидные маркеры (не круглые — это отличает от обычных графиков)

**Линии:**
- K: `#F97316` (оранжевый) — главное измерение, выделено
- A: `#60A5FA` (холодный синий) — `rgba(96,165,250,0.8)`
- Q: `#A78BFA` (фиолетовый) — `rgba(167,139,250,0.8)`
- S: `#34D399` (зелёный) — `rgba(52,211,153,0.8)`
- Толщина: 1.5px для вторичных, 2px для K (оранжевой)

**Область под линиями (area fill):**
- Тонкий gradient fill под каждой линией: от цвета линии (opacity 0.08) до transparent
- Это создаёт «вес» данных, не перегружая визуал

**Маркеры оценок:**
- Ромб 6×6px на каждой точке оценки
- Цвет соответствует линии
- Hover: увеличивается до 10×10px

**Ось X:**
- Подписи дат: IBM Plex Mono, 10px, `text-muted`
- Метки типа «Диагностика 1», «Диагностика 2» — над или под точками с вертикальной пунктирной линией

**Ось Y:**
- Горизонтальные сетевые линии: `rgba(255,255,255,0.04)`, очень тонкие
- Числовые метки: IBM Plex Mono, 10px, `text-muted`

**Аннотации (ключевые события):**
- Над определёнными точками — маленький тег: «Сессия проведена», «Процессы описаны»
- `caption` Inter, `bg-elevated`, border `border-subtle`, padding 3px 8px, border-radius 3px
- Соединяется с точкой тонкой вертикальной линией

**Легенда:** горизонтально над графиком. 4 элемента: цветная линия (16px) + буква + название. `body-sm`, `text-secondary`.

### Цвета и состояния

| Состояние | Визуал |
|----------|--------|
| Default | Все 4 линии видны, лёгкий area fill |
| Hover на маркер | Tooltip с датой, значениями всех 4 измерений в этой точке, delta от предыдущей оценки |
| Клик на легенду | Соответствующая линия выделяется, остальные затемняются до opacity 0.2 |

**Tooltip:**
- Заголовок: дата (IBM Plex Mono, 12px)
- 4 строки: цветная точка + буква + значение + delta (+0.4, -0.1) с цветом delta (зелёный/красный)
- Итого: общий score (bold)

### Анимации

1. **Появление:** линии рисуются слева направо — stroke-dashoffset animation, 1000ms ease-in-out, stagger 150ms между линиями. Area fill появляется вслед за линией.

2. **Hover на маркер:** 120ms ease-out, tooltip появляется над маркером с fade-in.

3. **Переключение легенды:** 200ms crossfade.

### Премиальность

- Вертикальные dotted линии от каждой точки к оси X — `rgba(255,255,255,0.06)`, создают структуру как у финансовых терминалов
- Размер: 600×280px на десктопе в мокапе
- Если компания показала рост — над графиком badge «Управляемость ↑ +18% за 3 мес.» с оранжевым акцентом

## 5. Preview интерфейса отчёта

### Задача
Показать, что диагностика выдаёт настоящий, профессиональный документ — не просто «ваш балл 67». Создать желание получить этот отчёт.

### Где используется
- Экран 6 («Диагностика») — правая колонка рядом с описанием
- Может появляться в Hero как фоновый элемент с blur

### Композиция

**Мокап браузерного окна или планшетного экрана** с интерфейсом отчёта внутри.

**Внешний контейнер (browser frame):**
- `bg-elevated`, border `border-default`, border-radius 12px
- Верхняя панель: 3 точки (close/minimize/maximize) в `text-muted` + строка «K-A-Q-S™ Report» по центру в IBM Plex Mono, 11px
- Высота панели: 36px, разделитель `border-subtle`
- Box-shadow: `0 32px 80px rgba(0,0,0,0.7)` — тяжёлая тень для ощущения «плавающего» над фоном

**Содержимое (интерфейс отчёта):**

Верхняя часть (header отчёта):
- Логотип K-A-Q-S™ слева, дата справа (IBM Plex Mono, 10px)
- Название компании (условное): «ACME Corp — Диагностика управляемости»
- Тонкий разделитель

Секция 1 — Общий показатель:
- Большое число в центре: «3.7 / 5.0» в IBM Plex Mono, 48px, `text-accent`
- Подпись: «Индекс управляемости» в `caption`, `text-muted`
- Под числом: горизонтальная шкала (track + fill), заполненная на 74%, `accent-primary`
- Справа от шкалы: уровень «СРЕДНИЙ» в `label`, bejdzh оранжевый

Секция 2 — Четыре показателя (2×2 мини-сетка):
- K: 4.1, A: 2.8, Q: 3.9, S: 3.2
- Каждый: буква IBM Plex Mono 20px, число 24px, мини-шкала 4px
- Критически низкий A: число `#EF4444`, шкала красная, иконка предупреждения

Секция 3 — Топ-3 риска (список):
- Пронумерованный список, каждый пункт: оранжевая точка + текст риска + категория-тег
- `body-sm`, `text-secondary`

Секция 4 — видна частично (обрезана снизу):
- Заголовок «Роадмап на 90 дней» — намекает что дальше есть ещё
- Первый пункт виден, остальные уходят за нижний край мокапа

### Цвета и состояния

- Только static мокап на лендинге
- Никаких hover-состояний внутри мокапа
- Сам контейнер (browser frame) может иметь hover: translateY(-4px), shadow усиливается

### Анимации

1. **Появление контейнера:** scale от 0.95 до 1.0 + opacity 0, 500ms ease-out. Задержка после появления текста слева.

2. **Счётчик числа (3.7):** count-up от 0 до 3.7 за 800ms. Шкала заполняется синхронно.

3. **Числа K/A/Q/S:** count-up stagger 120ms, всё в рамках 1000ms от старта.

4. **Критический A (2.8):** мигает один раз (opacity 1→0.4→1) через 1200ms после старта — привлекает внимание к проблеме.

### Премиальность

- Небольшой угловой поворот (perspective tilt): контейнер чуть наклонён вправо (rotateY(4deg) rotateX(2deg)) — создаёт ощущение 3D-объекта, а не скриншота
- Над мокапом — бейдж «PDF · 12 страниц» в `text-muted`, `bg-elevated`
- Небольшой blur + затемнение на нижних 20% мокапа — намёк что документ длиннее
- Слабый свет позади контейнера: `radial-gradient(ellipse, rgba(249,115,22,0.06), transparent 60%)`

## 6. System Architecture Visual — для Hero

### Задача
Создать ощущение масштаба и технологичности с первого взгляда. Человек должен почувствовать: «это серьёзная система», ещё до чтения текста.

### Где используется
- Экран 1 (Hero) — правая половина экрана на десктопе или фоновый элемент по центру
- На мобиле — упрощённая версия или полностью скрыт (текст выходит на первый план)

### Композиция

**Изометрическая схема из трёх слоёв** — как архитектурная диаграмма enterprise-системы, нарисованная линиями.

Не 3D-рендер. Не иллюстрация. Только SVG-линии, узлы, метки.

**Три горизонтальных слоя (снизу вверх):**

```
┌────────────────────────────────────────────┐  ← Слой 3: OUTPUT
│  [Отчёт]  [Роадмап]  [Риски]  [Динамика]  │
└────────────────────────────────────────────┘
              ↑  ↑  ↑  ↑
┌────────────────────────────────────────────┐  ← Слой 2: ENGINE (K-A-Q-S™)
│   [K]────────[A]────────[Q]────────[S]     │
│    │          │          │          │      │
│    └──────────┴──────────┴──────────┘      │
│                    [◉]                     │
└────────────────────────────────────────────┘
              ↑  ↑  ↑  ↑
┌────────────────────────────────────────────┐  ← Слой 1: INPUT
│  [Вопросы]  [Данные]  [Процессы]  [KPI]   │
└────────────────────────────────────────────┘
```

**Реализация (SVG):**

Слои:
- Каждый слой — прямоугольник с border `border-default` (0.75px), border-radius 8px, background `rgba(255,255,255,0.02)`
- Метка слоя: `label`, 10px, uppercase, `text-muted`, в левом верхнем углу слоя

Узлы внутри слоёв:
- Небольшие прямоугольники: `bg-elevated`, border `border-subtle`, border-radius 4px
- Текст внутри: IBM Plex Mono, 10px, `text-secondary`
- Для слоя ENGINE — узлы K/A/Q/S: border `border-accent`, letter `text-accent`, 12px, IBM Plex Mono

Центральный элемент ENGINE (◉):
- Круг 28px, border-accent, accent-glow background
- Символ K-A-Q-S™ или просто «KA QS» в 2 строки, IBM Plex Mono, 7px

Соединительные линии (вертикальные между слоями):
- 4 параллельные вертикальные линии от слоя INPUT к ENGINE, от ENGINE к OUTPUT
- `rgba(249,115,22,0.4)`, 1px
- Анимированные «бегущие пакеты данных» (движущиеся точки вдоль линий)

Горизонтальные связи внутри ENGINE:
- Тонкие `border-subtle` линии между K/A/Q/S
- Плюс диагональные линии от каждого к центральному ◉

**Внешнее обрамление:**
- Всё изображение в лёгкой изометрической перспективе (20–25° поворот, CSS transform: rotate(-5deg) skewX(-10deg) — упрощённая изометрия)
- Или полностью фронтальное (проще, не хуже)

### Цвета и состояния

- Static на десктопе — анимация только при появлении + idle
- На мобиле — упрощённая схема только ENGINE-слоя без INPUT/OUTPUT, без перспективы

### Анимации

1. **Появление (одноразовое):**
   - Слой 1 (INPUT) появляется первым: fade-in + translateY(10px), 400ms
   - Линии к ENGINE рисуются: 400ms, ease-out
   - Слой ENGINE: 300ms, с accent-glow вспышкой в момент появления (opacity 0→0.3→0.1)
   - Линии к OUTPUT рисуются: 400ms
   - Слой OUTPUT: fade-in, 300ms
   - Итого: ~1.8s от старта

2. **Idle (бесконечная):**
   - «Бегущие пакеты» по вертикальным линиям: маленькие светящиеся точки (3px) движутся от INPUT вверх к OUTPUT
   - `accent-primary`, opacity 0.8
   - Каждая линия: 1 точка каждые 2s, скорость — 1s на прохождение
   - Движение плавное, ease-in-out
   - Это единственное движение в idle — всё остальное статично

3. **Hover на узел (K/A/Q/S):**
   - Узел подсвечивается
   - Появляется тонкий tooltip с кратким описанием измерения

### Премиальность

- Изометрический вид (даже слабый) сразу поднимает восприятие — выглядит как enterprise architecture diagram
- Движущиеся пакеты данных создают ощущение живой системы без перегрузки
- Размер: 560×400px на десктопе — занимает правую половину hero (50% ширины)
- Очень слабый glow за всей схемой: `radial-gradient(ellipse 80% 60%, rgba(249,115,22,0.04), transparent)`
- Метки слоёв — как у Palantir Foundry: «DATA LAYER», «ANALYSIS ENGINE», «INTELLIGENCE OUTPUT»

## Технические ограничения (из CLAUDE.md)

- Все 6 элементов реализуются через SVG + CSS-анимации
- Никаких внешних библиотек (D3, Chart.js, Three.js)
- Radar Chart — SVG polygon, coordinates считаются через тригонометрию в JS
- Multi-line Chart — SVG path с polyline или cubic bezier
- Risk Map — SVG с positioned circles
- System Architecture — SVG с group, rect, line, circle
- Все числовые анимации — requestAnimationFrame или CSS counter (где возможно)
- IntersectionObserver запускает все анимации при попадании в viewport
